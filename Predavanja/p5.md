### `open`

Deklaracija u `fcntl.h` header-u:
``` c
int open(const char *pathname, int flags, ... /* mode_t mode */ );
```

Sistemski poziv `open` se koristi za otvoranje fajla.
Otvoranje fajla podrazumijeva uspostavljanje veze od fajl deskriptora do inode-a koji je asociran sa fajlom kojeg želimo otvoriti.
Opciono, može i kreirati fajl.

Povratna vrijednost `open` sistemkog poziva je novokreirani fajl deskriptor.
Ukoliko se desi greška prilikom izvršavanja, povratana vrijednost je `-1` i postavlja se globalna varijabla `errno` na error code koji predstavlja razlog greške.

Parametri ovog sistemskog poziva su:
- `pathname`
  - putanja do fajla u fajl sistemu koji želimo otvoriti
  - npr. `/home/root/foo.txt`
- `flags`
  - opcije sa kojima želimo otvoriti taj fajl
  - mora sadržavati jedno od: `O_RDONLY` (samo čitanje), `O_WRONLY` (samo pisanje) ili `O_RDWR` (i čitanje i pisanje) \*
  - ostale korisne opcije:
    - `O_APPEND` - prilikom svakog pisanja u fajl `offset` (u fajlu) se postavlja na kraj fajla
    - `O_CREAT` - ukoliko ne postoji fajl na `pathname` biti će kreiran
      - prilikom korištenja ove opcije kernel će gledati (opcioni) `mode` argument
      - ukoliko `mode` nije naveden kernel će uzeti šta god je na stack-u iznad argumenta za `flags` parametar (tretira se kao nedefinisano ponašanje)
      - ukoliko fajl već postoji ova opcija ništa ne radi, osim ako se koristi uz `O_EXCL` opciju; u tom slučaju vraća grešku
    - `O_TRUNC` - ako fajl postoji, postavlja njegovu veličinu na `0` (efektivno briše sadržaj fajla) \*\*
    - `O_NONBLOCK` - ako je moguće, osigurava da input i output operacije na fajl deskriptoru ne blokiraju
      - ovaj flag nema efekta na obične fajlove (i dalje će blokirati) \*\*\*
      - koristi se za socket-e, pipe i character uređaje (npr. terminal)
    - `O_CLOEXEC` - prilikom sistemskog poziva `exec` (`execve`) fajl se zatvara
      - sprječava kopiju fajl deskriptora
      - koristi se u procesima koji radi `fork` pa `exec` koje imaju fajlove otvorene sa eskaliranim (root) permisijama
      - ukoliko se ne koristi ova opcija child proces može dobiti fajl deskriptor za inode za koji nema permisije, budući da se permisije provjeravaju priliko otvoranja fajla, a ne prilikom korištenja
    - `O_SYNC` - forsira čekanje da svaka promjena (pisanje) fajla propagira do diska
- `... /* mode_t mode */`
  - opcioni parametar koji predstavlja file mode bite \*\*\*\*
  - primjenjuje pri kreiranju fajla (zajedno sa `umask` diktira permisije kreiranog fajla)
  - argument za ovaj parametar je 4-cifreni oktalni broj (ukoliko se koristi 3-cifreni prva cifra je `0`)
    - oktalni brojevi u C jeziku počinju sa `0` koja nije dio broja
    - prva cifra (s lijeva) diktira *set uid*, *set gid* i *sticky bit-e*
    - druga cifra diktira *owner* permisije
    - treća cifra diktira *group* permisije
    - četvrta cifra diktira *others* permisije
    - npr. `00664` ili `0664`:
      - *owner*: read i write
      - *group*: read i write
      - *others*: read
  - header `fcntl.h` ima definicije za neke permisije kao npr.:
    - `S_IRUSR` - owner read permisija (`0400`)
    - `S_IWUSR` - owner write permisija (`0200`)
    - `S_IRWXU` - owner read, write i execute permisija (`0700`)
    - `S_IWGRP` - group write permisija (`0020`)
    - `S_IXOTH` - others execute permisija (`0001`)
    - itd.

\*
*Ukoliko se ne navede neka od datih opcija, fajl se otvora u `O_RDONLY` modu. Ovo je rezultat toga što je `O_RDONLY` definisan kao `0` i toga što biti za mod pristupa fajlu se ne preklapaju sa drugim opcijama. U svakom slučaju, na ovo se ne treba oslonjati i treba eksplicitno navesti mod putem jedne od datih opcija.*

\*\*
*Ukoliko se zajedno koriste opcija `O_RDONLY` i `O_TRUNC` to je prema POSIX standardu nedefinisano ponašanje. Međutim, ono što Linux kernel uradi je da prebriše fajl i otvori ga u read only modu. Ovo se dešava zbog redoslijeda operacija koje kernel radi prilikom otvoranja fajla. `O_TRUNC` je operacija koja se dešava prilikom pristupa disku, prije nego što se kreira fajl u fajl tabeli. Zbog toga kernel briše sadržaj fajla iako je specificiran read only mode. Prije ove operacije se provjeravaju permisije za tu operaciju, pa nije moguće ovo uraditi ako korisnik nema write permisiju za taj fajl.*

\*\*\*
*Ukoliko stvarno postoji potreba za ne-blokirajućim čitanjem i pisanjem običnih fajlova, moderni pristup na Linux-u je koristeći [`io_uring` API](https://www.man7.org/linux/man-pages/man7/io_uring.7.html).*

\*\*\*\*
*Treći argument je ellipsis operator jer je ovo C API, a C ne podržava overloadanje funkcija, pa je ovo jedini način za specifikaciju opcionih parametara. Koristi se samo pri u slučajevima kada `open` kreira fajl dat sa `pathname`.*


**Happy code path (pretposavka da nema greški):**
- kernel prolazi kroz fajl sistem tražeći inode na putanji datoj sa `pathname`
- pronalazi inode
- zatim provjerava permisije inode-a, da li su u skladu sa modom korištenja datim `flags` argumentom
- provjera permisija prolazi
- učitava inode sa te putanje u memoriju, time dodaje novi entry u inode tabeli \
  (ukoliko inode već postoji u tabeli ne pravi se novi entry)
- kreira fajl u fajl tabeli
- kreira novi fajl deskriptor u fajl deskriptor tabeli
- vraća fajl deskriptor (indeks u tabeli)

**Glavni mogući problemi:**
- fajl ne postoji \
  Ukoliko fajl ne postoji, kernel gleda `flags` opcije.
  - ukoliko je navedena opcija `O_CREAT`
    - kernel će napraviti novi inode na disku tipa regularni fajl (`-` u `ls` ispisu)
    - permisije koje će taj fajl imati su diktirane sa (opcionim) `mode` argumentom
    - permisije se računaju kao `mode & ~umask` (efektivno uzima `mode` i gasi bite koji su set-ovani u `umask`)
  - ukoliko nije navedena opcija `O_CREAT`
    - kernel vraća grešku
- nedovoljne permisije \
  Ukoliko za traženi mod operacije (`flags` opcije) korisnik nema permisije `open` će vratiti grešku.

Provjeravanje permisija se vrši koristeći `euid` procesa koji poziva `open`.

Offset u fajlu je po default-u `0` osim ako se drugačije ne specificira (`flag`-ovima ili `mode`-om).

**Za više informacija pogledati [`man 2 open`](https://man7.org/linux/man-pages/man2/open.2.html).**

#### `creat`
Deklaracija u `fcntl.h` header-u:
``` c
int creat(const char *pathname, mode_t mode);
```

Za kreiranje fajla postoji sistemski poziv `creat`.
Ovaj sistemski poziv pravi fajl, postavlja mu veličinu na `0` (ako već postoji) i otvora ga isključivo za pisanje.
Povratna vrijednost je fajl deskriptor kreiranog/otvorenog fajla.
Poziv `creat` je ekvivalentno pozivu `open` sa `O_CREAT | O_WRONLY | O_TRUNC` `flag`-ovima.


### `close`

Deklaracija u `unistd.h` header-u:
``` c
int close(int fd);
```

Koristi se za zatvaranje fajl deskriptora.

Jedini parametar `close` sistemskog poziva je fajl deskriptor koji želimo zatvoriti.

Kernel u fajl deskriptor tabeli postavi pointer (na fajl u fajl tabeli) na `NULL`.
Provjeri da li file koristi još neki proces i ako zaključi da ne koristi, proglasi faj fajl slobodnim.
U tom slučaju provjerava i da li još neki fajl pokazuje na inode koji je asociran sa tim fajlom.
Ukoliko zaključi da ne pokazuje, proglašava i taj entry u inode tabeli slobodnim.

Ako procesi ne zatvore fajlove koje koriste može doći do popunjavanja fajl tabele u kojem slučaju se ne mogu otvoriti novi fajlovi.

Garbage collector ne pomaže sa ovim, pa je ovo problem koji imaju i jezici koji imaju garbage collector kao što su C#, Java, itd.

Ukoliko proces terminira OS će pozvati `close` na svim njegovim otvorenim fajlovima.

**Za više informacija pogledati [`man 2 close`](https://man7.org/linux/man-pages/man2/close.2.html).**


### `read`

Deklaracija u `unistd.h` header-u:
``` c
ssize_t read(int fd, void buf[.count], size_t count);
```

Koristi se za čitanje podataka iz fajl deskriptora.

Parametri ovog sistemskog poziva su:
- `fd` - fajl deskriptor iz kojeg želimo čitati
- `buf` - buffer u koji će kernel upisati pročitane podatke
  - buffer `buf` bi trebao minimalno moći pohraniti `count` bajti
- `count` - broj bajti koje želimo pročitati

Povratna vrijednost ovog sistemskog poziva je broj pročitanih bajti.
Ukoliko `open` vrati `0`, to je znak da više nema podataka koje je moguće pročitati (kraj fajla - `EOF`).
U slučaju greške vraća negativnu vrijednost (`-1`).
Konkretan error code se može pročitati iz `errno`.

`read` je vrlo često spor sistemski poziv, osim ako su podaci već u buffer cache-u.

`std::cin`, `getline` i slične metode čitanja podataka rade koristeći `read` sa standardnog ulaza.

Svaka `read` operacija pomijera offset u fajlu (u fajl tabeli) sa kojim je asociran dati fajl deskriptor.
`read` i `write` pomijeraju jedan te isti offset.

**Za više informacija pogledati [`man 2 read`](https://man7.org/linux/man-pages/man2/read.2.html).**


### `write`

Deklaracija u `unistd.h` header-u:
``` c
ssize_t write(int fd, const void buf[.count], size_t count);
```

Koristi se za pisanje podataka u fajl deskriptor.

Parametri ovog sistemskog poziva su:
- `fd` - fajl deskriptor u koji želimo pisati
- `buf` - buffer iz kojeg će kernel čitati podatke koje će upisivati
- `count` - broj bajti koje želimo upisati
  - vrijednost `count` treba da maksimalno bude veličina od `buf`

Povratna vrijednost ovog sistemskog poziva je broj bajti koji je upisan.
U slučaju greške vraća negativnu vrijednost (`-1`).
Konkretan error code se može pročitati iz `errno`.

`write` je tipično spori sistemski poziv (blokirajući), ali je brzi sistemski poziv (ne-blokirajući) ako je inode već u memoriji (buffer cache-u).

Svaka `write` operacija pomijera offset u fajlu (u fajl tabeli) sa kojim je asociran dati fajl deskriptor.
`read` i `write` pomijeraju jedan te isti offset.

**Za više informacija pogledati [`man 2 write`](https://man7.org/linux/man-pages/man2/write.2.html).**


### `lseek`

Deklaracija u `unistd.h` header-u:
``` c
off_t lseek(int fd, off_t offset, int whence);
```

Koristi se za pomijeranje fajl offset-a.

Parametri ovog sistemskog poziva su:
- `fd` - fajl deskriptor asociran sa fajlom kojem želimo pomjeriti offset
- `offset` - vrijednost za pomijeranje offset-a; način pomijeranja diktira `whence` argument
- `whence` - način pomijeranja offset-a
  - `SEEK_SET` - postavlja offset u fajlu na `offset` argument
  - `SEEK_CUR` - dodaje `offset` argument na trenutni offset
  - `SEEK_END` - postavlja offset u fajlu na kraj fajla plus `offset` argument

Prilikom korištenja `write` sistemskog poziva sa fajl deskriptorom koji je otvoren sa `O_APPEND` flag-om, `lseek` nema nikakvog efekta i podaci se uvijek pišu na kraj fajla.
Ovo nije slučaj za `read`, u tom slučaju se koristi offset koji je promijenjen pomoću `lseek`.

**Za više informacija pogledati [`man 2 lseek`](https://man7.org/linux/man-pages/man2/lseek.2.html).**

#### Sparsni fajl
Ukoliko je argument za `offset` veći od same veličine fajla, to je ok i fajl offset se pomijera na tu vrijednost.
Prilikom sljedećeg pisanja (`write` sa tim offset-om) veličina fajla se povećava na vrijednost offset-a plus količinu upisanih podataka.
Ovime se kreira **sparsni fajl**.

Efektivno, taj fajl ima "rupu" između orginalne veličine (prije `lseek` pa `write`) i `offset` argumenta pomoću kojeg je `lseek` pomjerio fajl offset.
Taj prazni dio ("rupa") se popunjava null karakterima (`\0`), ali na način da te nule zapravo ne zauzimaju prostor na disku.
Ovo je optimizacija koju podržaju moderni fajl sistemi (ext4, btrfs, XFS, NTFS, ...) u cilju čuvanja prostora na disku.
Ovo je veoma slično kako .bss sekcija radi u ELF fajlovima.
Veličina fajla na disku ostaje na orginalna veličina plus broj upisanih bajti.
Čitanjem iz tog praznog dijela ("rupe") pomoću `open` popunjava read buffer sa null karakterima (`\0`).

Čitanjem praznog dijela sparsnog fajla se zapravo ne čita fajl sa diska nego kernel samo popuni buffer null karakterima.
Promjena stvarne veličine fajla na disku se vrši tek pri pisanju `write` u prazni dio sparsnog fajla (ili na kraj).

Alati kao `ls` pokazuju logičku veličinu fajla (uključujući prazan dio sparsnog fajla).
Za čitanje fizičke veličine fajla (stvarne veličine na disku) može se koristiti alat `du`.

Primjer: \
`du -h sparsni_fajl.txt`

Za pregled veličina svih fajl sistema koji su mounted ("namontirani") može se koristiti alat `df`.

Primjer: \
`df -h`


### `fcntl`

Deklaracija u `fcntl.h` header-u:
``` c
int fcntl(int fd, int op, ... /* arg */ );
```

`fcntl` služi za izvršavanjem operacija na otvorenom fajl deskriptoru.
Uglavnom za čitanje i izmjenu fajl deskriptor flag-ova.

Parametri ovog sistemskog poziva su:
- `fd` - fajl deskriptor na kojem će se izvršiti operacija
- `op` - operacija koju želimo izvršiti
  - `F_GETFD` - `fcntl` vraća sve fajl deskriptor flag-ve (kao `int` u kojem pojedinačni biti predstavljaju flag-ove)
  - `F_SETFD` - postavlja fajl deskriptor flag-ove na vrijednost datu kao treći argument sistemskom pozivu (`arg`)
  - `F_GETFL` - vraća fajl status flag-ove (`O_RDONLY`, `O_CREAT`, `O_TRUNC`, itd.)
  - `F_SETFL` - postavlja fajl status flag-ove na vrijednost datu sa `arg`
    - moguće je promijeniti samo: `O_APPEND`, `O_ASYNC`, `O_DIRECT`, `O_NOATIME` i `O_NONBLOCK` flag-ove
    - ostali flag-ovi kao `O_RDWR`, `O_TRUC`, itd. se ignorišu
- `... /* arg */` - (opcioni) argument za operaciju (da li je potreban ili ne zavisi od operacije `op`)

**Za više informacija pogledati [`man 2 fcntl`](https://man7.org/linux/man-pages/man2/fcntl.2.html).**


### `stat`

Deklaracija u `sys/stat.h` header-u:
``` c
int stat(const char *restrict pathname,
         struct stat *restrict statbuf);
```

`stat` služi za čitanje informacija o fajlu.

Parametri ovog sistemskog poziva su:
- `pathname` - putanja do fajla čije informacije želimo pročitati
- `statbuf` - pointer na `struct stat` "objekat" u koji će se spremiti informacije o fajlu

Pregled `struct stat` strukture:
``` c
struct stat {
    dev_t     st_dev;     /* ID of device containing file */
    ino_t     st_ino;     /* Inode number */
    mode_t    st_mode;    /* File type and mode (permissions) */
    nlink_t   st_nlink;   /* Number of hard links */
    uid_t     st_uid;     /* User ID of owner */
    gid_t     st_gid;     /* Group ID of owner */
    dev_t     st_rdev;    /* Device ID (if special file) */
    off_t     st_size;    /* Total size, in bytes */
    blksize_t st_blksize; /* Block size for filesystem I/O */
    blkcnt_t  st_blocks;  /* Number of 512B blocks allocated */

    /* Timestamps */
    struct timespec st_atim;  /* Time of last access */
    struct timespec st_mtim;  /* Time of last modification */
    struct timespec st_ctim;  /* Time of last status change */
};
```

**Za više informacija pogledati [`man 2 stat`](https://man7.org/linux/man-pages/man2/stat.2.html).**

#### `lstat`

Deklaracija u `sys/stat.h` header-u:
``` c
int lstat(const char *restrict pathname,
          struct stat *restrict statbuf);
```

`lstat` je identičan `stat` sistemskom pozivu osim ako je `pathname` simbolički link.
U tom slučaju `lstat` neće ga dereferencirati nego će vartiti informacije o samom link-u.


#### `fstat`
Deklaracija u `sys/stat.h` header-u:
``` c
int fstat(int fd, struct stat *statbuf);
```

Sistemski poziv `fstat` radi istu stvar kao i `stat`, ali je sigurniji i nema problema sa race condition-ima.
U korištenju se razlikuju po tome što `stat` uzima putanju kao argument, a `fstat` uzima fajl deskriptor, što znači da fajl mora prethodno biti otvoren za korištenje `fstat`.
Zbog sigurnosti, `fstat` je preferirani sistemski poziv.

#### `fstatat`
Deklaracija u `sys/stat.h` header-u, ali je potreban i `fcntl.h` header ako se koriste `AT_*` konstante.

``` c
int fstatat(int dirfd, const char *restrict pathname,
            struct stat *restrict statbuf, int flags);
```

Problem koji `stat` i `fstat` imaju je što pri korištenju relativnih putanja u potpunosti se oslonja na `CWD` (Current Working Directory).
Ovo može stvarati problem ako neki drugi sistemski pozivi mijenjaju `CWD`, posebno ako se taj dio koda izvršava paralelno (više procesa ili thread-ove mijenja `CWD`).
Taj problem rješava `fstatat`.

Parametri ovog sistemskog poziva su:
- `dirfd` - fajl deskriptor za direktorij koji će se koristiti pri računanju relativnih putanja
  - može se koristit konstanta `AT_FDCWD` pri čemu će `dirfd` referencirati fajl deskriptor od `CWD`
- `pathname` - putanja do fajla (apsolutna ili relativna u odnosu na direktorij sa kojim je asociran fajl deskriptor `dirfd`)
  - u slučaju korištenja apsolune putanje `dirfd` se ignoriše
- `statbuf` - pointer na `struct stat` u koji će se spremiti informacije o fajlu
- `flags` - modifikatori ponašanja sistemskog poziva
  - `0` - bez modifikatora
  - `AT_EMPTY_PATH` - ukoliko je `pathname` prazan, operacija se vrši na fajlu koji je asociran sa fajl deskriptorom `dirfd`; u ovom slučaju `dirfd` ne mora pokazivati na direktorij, nego može pokazivati na proizvoljan tip fajla
  - `AT_SYMLINK_NOFOLLOW` - ako je `pathname` simbolički link, neće ga dereferencirati, nego će vratiti podatke o samom link-u


### `dup`

Deklaracija u `unistd.h` header-u:
``` c
int dup(int oldfd);
```

`dup` služi za duplikaciju fajl deskriptora.

Jedini parametar `dup` sistemskog poziva je fajl deskriptor koji želimo duplicirati.
Duplicirani fajl deskriptor će pokazivati na isti fajl (entry u fajl tabeli) kao i orginalni fajl deskriptor.
Duplicirani fajl deskriptor će zauzeti prvi slobodan entry u fajl deskriptor tabeli.

Povratna vrijednost je novi fajl deskriptor.

Problem sa ovim sistemskim pozivom je što proces nema nikakvu kontrolu o tome koju vrijednost će imati duplicirani fajl deskriptor.

Recimo da želimo zatvoriti standardni izlaz i otvoriti novi fajl sa fajl deskriptorom `1` koji će predstavljati standardni izlaz.
Korisimo `close(STDOUT_FILENO)`, zatim otvorimo neki fajl sa `open("test.txt", O_WRONLY)`.
Moguće je da se dešava paralelno izvršavanje nekog koda koji između poziva `close` i `open` otvori neki fajl.
U tom slučaju taj neki fajl će se otvoriti sa fajl deskriptorom `1`, a `test.txt` će se tovoriti na prvom sljedećem slobodnom mjestu.
Ovo je ogroman problem jer je sada naš standardni izlaz u neki fajl za kojeg mi uopće ne znamo šta je, a i onaj ko je otvorio taj fajl ima standardni izlaz preusmjeran u taj tajl, što vjerovatno ne očekuje.

Ovaj problem rješava sistemski poziv `dup2`.

#### `dup2`
Deklaracija u `unistd.h` header-u:
``` c
int dup2(int oldfd, int newfd);
```

`dup2` radi isto što i `dup` (duplicira fajl deskriptor `oldfd`), ali umjesto da duplicirani fajl deskriptor smjesti na prvo slobodno mjesto, smjesti ga na `newfd`.
Ukoliko je na `newfd` već postojao fajl deskriptor, zatvorit će se (kernel poziva `close`) i zamijeniti sa `oldfd`.
Ukoliko su `oldfd` i `newfd` isti sistemski poziv neće uraditi ništa i samo će vratiti taj fajl deskriptor.

`dup2` sve radi atomski (unutar kernela, atomski iz perspektive user procesa), pa nema prostora za race condition-om koji nastaje korištenjem `close` i `dup`.

*Postoji i `dup3` koji radi kao i `dup2`, ali njega nismo nigdje spominjali. `dup3` omogućava atomsko postavljanje flag-ova (specifično `O_CLOEXEC`). Postoji jer `dup2` može dovesti drugoj vrsti race condition-a. Ukoliko je potrebno postaviti `O_CLOEXEC` flag, nakon `dup2` to se mora uraditi sa `fcntl` što je dva sistemska poziva i samim time nije atomski. `dup3` rješava ovaj problem. Dodatno, `dup3` će javiti grešku ako su `oldfd` i `newfd` isti.*


### bash

Bash ima par binarnih operatora za redirekciju ulaza i izlaza procesa sa terminala u fajlove:
- operator `>` - redirekcija standardnog izlaza u fajl
- operator `>>` - redirekcija standardnog izlaz u fajl
- operator `<` - redirekcija standardnog ulaza na fajl
- operator `<<` - redirekcija standardnog ulaza na višelinijski string sa delimiterom dat u terminalu

Prilikom korištenja svakog od navedenih operatora bash uradi `fork`, poziva sistemske pozive koje rade redirekciju i zatim `exec`.

#### operator `>`
Operator `>` preusmjerava standardni izlaz procesa kojeg pokreće komanda koja se nalazi sa lijeve strane operatora `>` u fajl koji se nalazi sa desne strane operatora `>`.

Primjer korištenja: \
`echo "Test" > file.txt`

`echo` ispisuje sve argumente na standardni izlaz.
Operator `>` preusmjerava standardni izlaz procesa kreiranog sa `echo "Test"` u fajl `file.txt`.
Ukoliko taj fajl ne postoji bash će ga kreirati.
Ukoliko je taj fajl već imao neki sadržaj, taj sadržaj će biti prepisan.

Efektivno, bash radi sljedeće:
``` c
int fd = open("file.txt", O_WRONLY | O_CREAT | O_TRUNC, 0666);
dup2(fd, STDOUT_FILENO);
close(fd);
```

Sistemski pozivi:
- `open` sa flag-ovima `O_WRONLY | O_CREAT | O_TRUNC` i `mode`-om `0666`
  - `O_WRONLY` - zato jer će se samo pisati u fajl (standardni izlaz)
  - `O_CREAT` - kreira fajl ako ne postoji
  - `O_TRUNC` - briše sadržaj fajla ako postoji
  - `mode` `0666` - pokušava dati i read i write permisije svima, kombinuje se sa `umask` koji ih filtrira i daje stvarne permisije fajla
- `dup2` fajl deskriptor a kreiranog sa `open` na `STDOUT_FILENO` (standardni izlaz, fajl deskriptor `1`)
- `close` na fajl deskriptor kreiran sa `open` (ne treba više jer fajl deskriptor `1` pokazuje na taj fajl)

#### operator `>>`
Operator `>>` radi isto što i operator `>`, ali umejsto `O_TRUNC` koristi `O_APPEND`.

Primjer korištenja: \
`echo "Test" >> file.txt`

Efektivno, bash radi sljedeće:
``` c
int fd = open("file.txt", O_WRONLY | O_CREAT | O_APPEND, 0666);
dup2(fd, STDOUT_FILENO);
close(fd);
```

Zbog `O_APPEND` svaki ispis na standardni izlaz će pisati na kraj datog fajla i neće prebrisati već postojeći sadržaj.

#### operator `<`
Operator `<` preusmjerava standardni ulaz procesa kojeg pokreće komanda koja se nalazi sa lijeve strane operatora `<` na fajl koji se nalazi sa desne strane operatora `<`.

Primjer korištenja: \
`cat < file.txt`

`cat` čita sa standardnog ulaza i ispisuje iste podatke na standardni izlaz.
Operator `<` preusmjerava standardni ulaz procesa kreiranog sa `cat` na fajl `file.txt`.
Ukoliko taj fajl ne postoji bash će javiti grešku.

Efektivno, bash radi sljedeće:
``` c
int fd = open("file.txt", O_RDONLY);
dup2(fd, STDIN_FILENO);
close(fd);
```

Prilikom sistemskog poziva `open` koristi se `O_RDONLY` mode jer će proces samo čitati iz fajla.
Kao target fajl deskriptor u `dup2` se koristi `STDIN_FILENO` (standardni ulaz, fajl deskriptor `0`).

#### operator `<<`
Operator `<<` je najkompleksniji operator i razlikuje se od ostala tri.
Detalji kako radi neće biti opisani ovdje jer koristi pipe-ove koji još nisu pređeni.
Samo će se opisati kako se koristi.

Primjer korištenja:
``` bash
cat << EOF
Hello world!
Druga linija
Sve ovo ide u cat
EOF ukljucujuci ovo
i ovo EOF
i ovo
EOF
```

Sa lijeve strane operatora `<<` treba biti komanda.

Sa desne strane operatora `<<` treba biti proizvoljan delimiter (tipično se koristi `EOF`, ali nema ništa posebno sa njim, nema nikakvo posebno značenje, može se koristiti bilo šta, npr. `foobar`).

Nakon delimitera, u novim linijama se unosi tekst koji želimo proslijediti komandi.
Čitanje se vrši sve do linije koja se sastoji od delimitera i ničega više (čak ni prazno mjesto ` `).

Primjer sa drugim delimiterom:
``` bash
cat << foobar
Test1
Test2
Test3
foobar
```

#### Kombinacija operatora
Moguće je i kombinovati ove operatore.

Primjer: \
`cat < foo.txt > bar.txt`

Standardni ulaz od `cat` se preusmjerava na `foo.txt`, a standardni izlaz na `bar.txt`.
Efektivno se ovime fajl `foo.txt` kopira u `bar.txt` pri čemu se sadržaj `bar.txt` prepisuje (briše).

Primjer: \
`cat < foo.txt >> bar.txt`

Isto kao i prethodni primjer, ali sadržaj `bar.txt` se ne prepisuje, nego se sadržaj `foo.txt` piše na kraj fajla `bar.txt`.
