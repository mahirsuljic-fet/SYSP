### Login

Pri pokretanju terminala se pokreće `login` proces i to u ime `root` user-a.
`login` proces traži username i password od korisnika.
Kada korisnik unese username i password, `login` gleda u `/etc/passwd` da vidi da li postoji taj user.
Zatim gleda u `/etc/shadow` i provjerava da li je uneseno password tačan.
Password u `/etc/shadow` je hash-iran, pa prije upoređivanja je potrebno hash-irati i uneseni password.
Ako je sve ok, pokreće se shell kojeg `login` pronađe u `/etc/passwd` za tog korisnika (većinom `/bin/bash`).
`$CWD` od shell procesa je `$HOME` koji se pročita iz `/etc/passwd` za ulogovanog korisnika.
`login` se izvršava u ime `root` user-a, a shell treba da se izvršava u ime user-a koji se ulogovao.

Ime korisnika koji je trenutno ulogovan se može pročitati iz `/etc/hostname` ili `$USER` environment varijable.

Environment varijabla `$PS1` se ispisuje u terminal kada terminal čeka unos od korisnika i tipično ima format sličan `username@hostname:current_directory_path$ `.

Grupe su definisane u `/etc/group`.


### Shell komande

Komande se generalno mogu svrstati u dvije kategorije:
- shell builtin
- eksterni programi

**Shell builtin** su komande koji sam shell izvršava interpretiranjem unesenog teksta bez potrebe korištenja eksternih alata.
Ove komande se mogu koristiti čak i ako je `$PATH` environment varijabla prazna.

**Eksterni programi** su programi koji se nalaze na fajl sistemu, njih shell traži u direktorijama koje su navedene u `$PATH` environment varijabli i izvršavaju se korištenjem `fork` i `exec` sistemskih poziva.

Vrsta komande se može saznati korištenjem `type` komande.
Npr. pomoću `type echo` se može vidjeti da je `echo` komanda shell builtin, a pomoću `type cat` se može vidjeti da je `cat` eksterni program i gdje se on nalazi u fajl sistemu.

Dakle shell izvršava eksterne komande pomoću sistemskog poziva `exec`, ali mora prvo pozvati `fork`, pa tek onda `exec`.
Ako bi odma pozvao `exec`, jedina instanca shell-a bi se zamijenila sa instancom traženos programa (npr. `ls`) i prilikom terminacije tog procesa cijeli terminal bi terminirao\*.
Pozivanjem `fork` se dobija child shell process koji izvršava `exec`, a parent čeka u sistemskom pozivu `wait` da child proces terminira, nakon čega se kontrola ponovo predaje parent procesu. \
**Pravi terminal (kao računar) bi se ugasio, a na modernim računarima ovo znači zatvoranje terminal prozora, odnosno terminacija shell procesa.*

Ovo se može i demonstrirati.

#### Exec only primjer
Postoji `exec` bash builtin koji je wrapper oko `exec` sistemskog poziva.
Budući da je u pitanju shell (bash) builtin, korištenjem ove komande neće se desiti `fork`.
Izvršavanjem `exec ls` u terminal se ispisuje output `ls` komande i terminal se gasi.
Ovo se može još bolje vidjeti korištenjem `sleep` komande koja jednostavno čeka dati broj sekundi pa zatim terminira.
Izvršavanjem `exec sleep 2` terminal čeka `2` sekunde i zatim terminira.

#### Fork primjer
Unutar terminala se može izvršiti komanda `bash`.
`bash` je eksterni program, pa će se desiti `fork`.
Terminal će izgledati i ponašati se isto nakon izvršenja ove komande jer se izvršava isti program kao i prije nje, ali ne i isti proces.
Izvršava se child proces, a parent proces čeka u `wait` sistemskom pozivu.
Može se promijeniti direktorija sa npr. `cd /` i sa komandom `pwd` ili `echo $PWD` se ispisuje trenutna putanja koja će zaista biti promijenjena.
Ako sada izvršimo `exit` ili uradimo `CTRL + D`, trenutni (child) proces će se terminirati.
Kontrola se vraća parent shell procesu koji se budi iz `wait`.
Ovo je evidentno time što se nije cijeli terminal ugasio i ako se izvrši `pwd` ili `echo $PWD`, putanja će biti ista kao i prije `cd /` u child procesu, jer putanja parent shell-a nije promijenjena.


### Permisije

#### Vrste permisija
Svaki `inode` (fajl, direktorij, uređaj, ...) ima određene bite koji govore koji korisnici imaju permisije za oređene operacije nad `inode`-om.
Postoje po tri bita za vlasnika fajla (owner), grupu (group) i ostale (others).
Ta tri bita su `r` (Read), `w` (Write) i `x` (eXecute).

Pomenuti bit-i se mogu vidjeti izvršenjem komande `ls -l` gdje su prikazani u formatu `Trwxrwxrwx` gdje je `T` tip `inode`-a (`-` je file, `d` direktorij, itd.), prva tri `rwx` se odnose na owner-a, druga tri na grupu i zadnja tri na ostale korisnike.

Prilikom pristupa `inode`-u vrši se autorizacija koja je detaljnije objašnjena dalje u ovom fajlu.
Ukratko, ako korisnik želi pročitati sadržaj fajla i vlasnik je fajla, provjerava se prvi `r` bit.
Dakle, ako fajl ima `-rwx------` owner može pročitati fajl jer je `r` set-ovan, ali ako bi imali `--wxrwxrwx` owner ne može pročitati fajl.
Ako korisnik nije owner, provjerava se da li je u istoj grupi kao i fajl.
Ako jest provjeravaju se druga tri `rwx` bit-a.
Ako nije ni u istoj grupi provjeravaju se zadnja tri `rwx` bit-a.

#### Promjena permisija
Za promjenu permisija koristi se utility `chmod` koji je wrapper za istoimeni sistemski poziv, i može ga koristiti samo vlasnik fajla ili `root` user.

Recimo da želimo dodati execute permission svima za fajl `foobar`, to se radi sa `chmod +x foobar`.

Generalno, kažemo koji bit želimo promijeniti (`r` - read, `w` - write, `x` - execute), kome (`u` - owner, `g` - group, `o` - others) i kako (`+` - postavi, `-` - ukloni).

Ispod je još par primjera korištenja `chmod`:
- `chmod u+x foobar` - dodaj execute permission za owner-a
- `chmod g+x foobar` - dodaj execute permission za grupu
- `chmod o+x foobar` - dodaj execute permission za sve ostale
- `chmod u+w foobar` - dodaj write permission za owner-a
- `chmod o-w foobar` - oduzni write permission za ostale

Moguće je i koristiti kombinacije:
- `chmod ug+x foobar` - dodaj execute permission za owner-a i grupu
- `chmod u+rx foobar` - dodaj execute i read permission za owner-a
- `chmod go+wx foobar` - oduzmi execute i write permission za grupu i ostale

Moguće je i svih 9 bit-a postaviti odjednom koristeći trocifreni oktalni broj, gdje se prva cifra odnosi na owner-a, druga na grupu i treća na ostale:
- `chmod 600 ./foobar` - postavi rw- za user --- za group --- za ostale
- `chmod 777 ./foobar` - postavi rwx za user rwx za group rwx za ostale

#### Promjena owner-a i grupe
Slično kao i za promjenu permisija, postoje i utility-i za promjenu owner-a i grupe, a to su `chown` i `chgrp` respektivno.

Međutim, ove komande su restriktivnije.
`chmod` može koristiti samo `root` user.
`chgrp` može koristiti vlasnik fajla, ali može promijeniti grupu samo u grupu čiji je i on sam član, dok `root` user može bezuslovno mijenjati grupu.

Sintaksa je:
- `sudo chown new_owner file_name` (potreban `sudo` zbog `root` ograničenja)
- `chgrp new_group file_name`

Kako `sudo` radi će biti objašnjeno kasnije u fajlu u dijelu za eskalaciju privilegija.

Moguće je istovremeno promijeniti owner-a i grupu pomoću `chown` utility-a koristeći sintaksu `sudo chown new_user:new_group file_name`.

#### Izvršavanje fajlova
Generalno postoje dvije vrste izvršnih fajlova:
- ELF (binarni)
- skripte

Ukoliko je fajl u ELF formatu, prilikom izvršenja OS ga učitaje u memoriju pomoću loader-a i pravi novi proces.

Ako nije ELF shell (bash) pokušava interpretirat fajl liniju po liniju.
Po default-u, ako se koristi bash shell, fajl se pokušava interpretirati kao bash programski jezik.
Međutim, postoji poseban komentar koji bash drugačije interpetira.
Komentari u bash programskom jeziku počinju sa `#`.
Ako je komentar prva linija u fajlu i počinje sa `#!`, takav komentar ima posebno značenje i naziva se **shebang** (čita se *šabeng*).
Shebang je uputa shell-u kako da interpretira dati fajl (skriptu).
Nakon `#!` slijedi putanja do programa koji će interpretirati ostatak fajla.
Npr. `#!/usr/bin/python` poziva python koji će interpretirati ostatak skripte.

Međutim `#!/usr/bin/python` ima problem.
Oslonja se u potpunosti na to da se python nalazi na putanji `/usr/bin/python`.
Ako sistem na kojem se izvršava skripta nema python na toj putanji, nego na nekoj drugoj (ili nema nikako) desit će se greška.
Ovo se zaobilazi koristeći `env` kao `#!/bin/env python` ili još bolje `#!/usr/bin/env python`.
Ovaj poboljšani shebang poziva `env` sa argumentom `python`.
Ono što će se desiti je da će `env` pregledati sve direktorije u `$PATH` environment varijabli tražeći `python`.
Ako ga nađe uradit će `exec` i interpretirati ostatak skripte kao python.
Ako ga ne nađe to znači da ne postoji `python` na path-u, te javlja grešku.

### Permisije na direktorijima

Permisije se odnose na `inode`.
U slučaju direktorije, operacija *read* se odnosi na čitanje linkova direktorije, operacija *write* se odnosi na promjenu linkova direktorije, a operacija *execute* se odnosi na pretraživanje direktorije.

Ovo znači da ako se oduzme read permisija na direktoriju nije moguće pročitati linkove koje ona ima, ali ako ima write permisiju, iako nije moguće pročitati linkove moguće ih je izbrisati.
Moguće je i čitati, pisati i izvršavati sadržaj fajlova unutar te direktorije (uzimajući u obzir permisije na fajlovima) iako sama direktorija nema read permisiju.

Primjer:
``` bash
mkdir test
ls test                 # OK
chmod -r test
ls test                 # Error jer nema read permisije
touch test/foo          # OK jer direktorij ima write permisiju
echo "123" > test/foo   # OK jer fajl ima write permisiju
cat test/foo            # OK jer fajl ima read permisiju
cd test                 # OK jer direktorij ima execute permisiju
ls                      # Error jer direktorij . (test) nema read permisije
cd ..
rm -r test              # Error jer u test ima fajl foo koji je potrebno
                        #   izbrisati prije brisanja samog test direktorija,
                        #   a budući da ne možemo pročitati sadržaj test-a,
                        #   ne možemo ni rekurzivno izbrisati sve unutar njega
rm test/foo             # OK jer direktorij ima write permission
rm -r test              # OK jer je direktorij sada prazan
```

Bez execute permisije ne može se uraditi `cd` niti pretraživati linkovi direktorije.
Pretraživanje linkova podrazumijeva pristup inode-ima na koje linkovi pokazuju.
Ovo također podrazumijeva kreiranje, modifijaciju i brisanje link-ova i fajlova na koje linkovi pokazuju.

Primjer:
``` bash
mkdir test
touch test/foo
touch test/bar
ls test
chmod -x test
ls test         # OK jer direktorij ima read permision,
                #   ali moguće je vidjeti samo imena fajlova (linkove)
touch test/tar  # Error jer ovo zahtijeva kreiranje novog linka i novog inode-a,
                #   a prilikom kreiranja novog linka se update-a
                #   i sam inode direktorije (npr. access timestamp)
ln test/foo hl  # Error iz istih razloga kao i touch
rm test/foo     # Error is istih razloga
ls -l test      # Error ili se ne dobijaju nikakve vrijednosti (većinom bude znak ?)
rm -r           # Error za svaki fajl
chmod +x test
rm -rf test
```


### Autorizacija

Zbog sigurnosti, u UNIX sisteme dodan je način provjere da li dati korisnik smije izvršiti neki sistemski poziv koji se naziva **autorizacija**.

Postoje tri vrste user i group ID-ova koji su bitni za autorizaciju:
- real (`ruid`/`rgid` - Real User/Group ID)
- effective (`euid`/`egid` - Effective User/Group ID)
- saved (`suid`/`sgid` - Saved User/Group ID)

#### Proces autorizacije
Prilikom autorizacije OS provjera da li user koji je pozvao sistemski poziv ima dozvolu da odradi tu operaciju na nekom objektu koji je cilj sistemskog poziva (npr. fajl, direktorij, uređaj, itd.).
Ukoliko user ima permisiju, sistemski poziv se izvršava u njegovo ime.

Cijeli proces autorizacije je sljedeći.

Korisnik pokrene neki program pri čemu se pravi proces.
Taj proces ima postavljene `ruid`, `euid` i `suid` na `uid` (user ID) korisnika koji izvršava program i `rgid`, `egid` i `sgid` na primarni `gid` (primarni group ID).
Kada program dođe do sistemskog poziva kernel vrši autorizaciju.
Za autorizaciju se koriste efektivni ID-ovi (`euid` i `egid`).

OS provjerava da li je `euid` isti kao `uid` vlasnika objekta koji je cilj sistemskog poziva.
Zbog lakše analize pretpostavimo da je cilj sistemskog poziva modifikacija sadržaja nekog fajla `foobar`.
Ako je `euid` vlasnik fajla, tada OS provjerava owner permission bit-e fajla `foobar` i u ovom slučaju konkretno provjerava `w` (write) bit jer je cilj modifikacija fajla.

Ako korisnik ima write permisiju sistemski poziv se izvršava.
Ako korisnik nema write permisiju OS zatim provjerava da li je `egid` isti kao `gid` grupe kojoj pripada fajl `foobar`\*.

Ako jest, provjeravaju se group permission bit-i fajla `foobar`, u ovom slučaju `w` bit.
Ako grupa ima write permisiju, OS izvršava sistemski poziv.
Ako grupa nema permisiju, OS provjerava other users permission bite fajla `foobar` (`w` bit).
Ako ostali korisnici imaju write permission sistemski poziv se izvršava, a ako nemaju onda OS prijavljuje grešku.

\**Pored primarne grupe provjeravaju se i suplementarne (sporedne) grupe.*

Ukoliko je korisnik u ime kojeg se zahtijeva sistemski poziv `root` user (`euid = 0`), sistemski poziv se bezuslovno izvršava (efektivno se preskače proces autorizacije).

Dalja analiza će se fokusirati na user ID-ove, ali isto vrijedi i za group ID-ove za slučaj kada se koriste permisije grupe.

`ruid` se može postaviti na `euid` ili na `suid`. \
`euid` se može postaviti na `ruid` ili na `suid`. \
`suid` se može postaviti na `ruid` ili na `euid`.

`root` user može postaviti `ruid`, `euid` i `suid` na bilo šta.


### Eskalacija privilegija

#### `ps` utility
Za dodatnu analizu može poslužiti `ps` utility.
`ps` utility se koristi za ispis informacija o aktivnim procesima.

Najčešće korištena komanda je `ps aux` koja ispisuje sve aktivne procese u user-oriented formatu, uključujući i procese drugih korisnika.

Korisni flag-ovi:
- `-A` svi procesi
- `a` svi procesi sa terminalom, ulključujući druge korisnike
- 'u' user-oriented format (čitljivije i više korisnih informacija)
- `-o ...` format ispisa

Dakle, sa `ps -A u` se ispisuje sve procese u korisnom formatu.

Za analizu autorizacije korisna je komanda `ps a -o user,pid,ruid,euid,suid,comm` koja ispisuje korisnika u ime kojeg se izvršava proces, PID procesa, `ruid`, `euid` i `suid` procesa kao i komandu koja je korištena za pokretanje procesa.

Ako se otvore dva terminala, u prvom izvrši `sleep 10`, a u drugom prethodna `ps` komanda sa formatiranjem dobit će se nešto slično kao `user      12345  1000  1000  1000 sleep` što govori da se izvršava `sleep` u ime korisnika `user` sa PID `12345` i da su svi uid-i postavljeni na `1000` (korisnik `user`).

#### Mehanizam eskalacije privilegija
Pored glavnih bit-a za permisije postoje i:
- set uid bit
- set gid bit
- sticky bit

U nastavku će se govoriti o set uid bit-u.

**Set uid bit** se koristi za pokretanje nekog programa u ime drugog korisnika.
Ovim mehanizmom je moguća eskalacija privilegija gdje npr. bilo koji korisnik može izvršiti neki program kao root ili neki drugi korisnik.

`ls` utility ima posebnu reprezenaciju za ovaj bit (set uid):
- set uid bit + execute = `s`
- set uid bit ali ne i execute = `S`

Ovo se može vidjeti na programima kao što je `passwd` koji se nalazi u `/usr/bin/passwd` (može se uraditi `type passwd` da se provjeri putanja).
Utility `passwd` se koristi za promjenu korisničkog password-a.

Fajl `passwd` bi trebao imati permisije slične kao `-rwsr-xr-x`, gdje `s` na mjesto `x` bit-a od owner-a označava da je set-ovan i execute i set uid bit.
Ako bi uklonili execute permisiju prikaz bi bio `-rwSr-xr-x`, a ako bi uklonili set uid permisiju, a ostavili execute permisiju prikaz bi bio `-rwxr-xr-x`.

Set uid bit ima efekta na `exec` syscall.
Prilikom izvršavanja fajla koji ima set uid bit postavljen OS postavlja `euid` i `suid` na `uid` od usera koji je owner tog fajla (npr. ako je root owner postavit će ih na `0`), dok je `ruid` user ID (`uid`) od korisnika koji je zapravo izvršio fajl (npr. `1000`).

`ruid` i `suid` se koristi za eskalaciju i deeskalaciju privilecija u toku izvršavanja procesa, a `euid` se koristi za provjeru privilegija kao što je objašnjeno u dijelu za autorizaciju.

`ruid` i `suid` su potrebni za prebacivanje između dva nivoa privilegija.
Recimo da je `ruid` postavljen na `1000`, a `suid` na `0`.
Budući da je `euid` moguće postaviti na `ruid` ili `suid`, program može proizvoljno mijenjati nivo privilegija između nivoa privilegija aktivnog korisnika i root-a postavljanjem `euid` na `ruid` ili `suid` respektivno.

Na isti način radi i `sudo`.
Ako se uradi komanda `ls -l $(type -p sudo | xargs realpath)` može se vidjeti da `sudo` ima set-ovan set uid bit.

Ova eskalacija privilegija se može vidjeti koristeći `ps` utility.
Potrebno je otvoriti dva terminala.
U jednom izvršiti `passwd`, a u drugom `ps a -o user,pid,ruid,euid,suid,comm`.
Output bi trebao biti sličan kao:
```
USER         PID  RUID  EUID  SUID COMMAND
                    ...
root       12345  1000     0     0 passwd
                    ...
```
Ovdje se jasno vidi da se komanda izvršava sa `ruid` `1000` (korisnik koji je pokrenuo komandu), da je `euid` `0` (treuntno se izvršava sa root privilegijama) i da je `suid` `0` (jer je owner fajla root sa `uid` `0`).
