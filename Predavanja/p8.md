Signal handleri se izvrsavaju u user space-u.

U `proc` sturkturu od procesa imaju tri polja koja se ticu procesiranja signala:
- vektori bita `pending`
- vektori bita `blocked`
- signal handleri (dispozicija) (?)

Svaki bit u vektoru je zaslluzen za jedan signal.

Ako je signal pending i nije blocked to znaci da treba biti procesiran.

Child proces dobija dispoziciju i `blocked` status, ali ne dobija `pending` status od parent-a prilikom `fork`-a.
Pozivom `exec` resetuje i dispoziciju i `blocked` status.

Kernel signal procesira tako da set-uje bit u `pending` vektoru u `proc` strukturi od tog procesa.

Dakle, ako dva procesa posalju trecem procesu jedan te isti signal u isto vrijeme, on ce biti procesiran samo jednom.


***Podsjetiti profesora da predje handling singala uz threadove kada budemo to radili.***


Signal se procesira kada kernelr schedule-a proces.

Da bi procesirali signal moramo biti u runnable stanju.
Dakle, ako smo u waiting stanju necemo procesirat.
Kernel MOZE (ali ne nuzno mora) probuditi proces dok je waiting stanju.
Ako se signal tretira dok je proces u waiting stanju, kernel ce vratit neuspijeh kao rezultat tog sistemskog poziva.

Izvrsavanje handlera moze biti prekinuto timer interuptom.

`sigset_t`    - set signala \
`sigaddset`   - dodaje signal u signal set \
`sigdelset`   - brise signal iz signal seta \
`sigismember` - projverava da li je signal dio singal seta \
`sigprocmask` - blokira ili odblokira signale iz signal set-a

`signal` ne postavlja blocked bit, resetuje signal handler nakon izvrsavanjea handlera, pa se smatra legacy API-om

Moderna verzija `signal` je `sigaction`.

Znamo `fork` multitasking model, naucit cemo threading i IO multipleksiranje.

`fork` model: \
Kada child proces terminira prelazi u zombie stanje.
Pri tome se salje `SIGCHLD` signal parent procesu.
Parent treba da ocisti njegovu memoriju.
To radi pomocu `wait` syscall-a.
Medjutim on je blokirajuci.

`sa_flags` - dodatke postavke za signal koje govorimo kernelu

`SIGCHLD` se salje i na suspendiranje.
Ako to ne zelimo mozemo dodati `SA_NOCLDSTOP` flag u `sa_flags`.

`SA_RESTART` - kada se probuci iz blocking syscall nakon sto je signal handler bio interruptan, kernel vraca proces u `WAITING` stanje ("restartuje" syscall)

Svaki proces koji poziva `fork` bi trebao imati signal handler koji poziva neblokirajuci `waitpid` kako bi oslobodio memoriju child procesa.


## Pipes

Pipe ne egzistira na fajl sistemu, ali ima fajl deskriptor.
Fifo egzistira na fajl sistemu.

Pipe se koristi za komunikaciju izmedju procesa u nasljednoj hijerarhiji (parent-child).
Fifo se koristi za komunikaciju izmedju proizvoljnih procesa.

Pipe je data struktura na kernel strani koja podrzaje push i pop operacije.
Efektivno je kao queue.

Pipe ima kapacitet, po defaultu kernel kaze koliki je taj kapacitet, ali moguce ga je konfiturisati.
Ukoliko se pipe napuni, proces ide u `WAITING` stanje.

Za kreiranje pipe-a koristi se syscall `pipe` koji nam "vraca" dva fajl deskriptora, jedan u koji pisemo, a drugi iz kojeg citamo.
Taj pipe se kreira unutar tog procesa.
Medjutim, moze se uraditi fork pri cemu se kopiraju ti fajl deskriptori.
Nakon fork-a parent i child trebaju da zatvore (syscall `close`) fajl deskriptor koji nece koristiti (npr. parent zatvori za citanje, a child zatvori za pisanje).

Bash ima operator `|` koji pravi pipe.
Uradi `fork`, napravi pipe, prepravi fajl deskriptore i izvrsi `exec`.
